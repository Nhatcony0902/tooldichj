# Wiki Conventions — Filenames, Links, Structure, Frontmatter

## Filename rules

GitHub wiki pages live in the wiki repo root — no subdirectories (GitHub flattens them at render time).

| Rule | Example (good) | Example (bad) |
|---|---|---|
| Hyphen-separated words | `Mechanic-Registry.md` | `Mechanic Registry.md`, `mechanic_registry.md` |
| PascalCase per segment | `Combat-System.md` | `combat-system.md`, `CombatSystem.md` |
| `.md` extension | `Home.md` | `Home`, `Home.markdown` |
| Reserved: leading `_` for sidebar/footer | `_Sidebar.md`, `_Footer.md` | `Sidebar.md`, `footer.md` |
| No subdirectories | `Boss-Mechanics.md` | `mechanics/Boss.md` |

The validator rejects violations. `fix-page-names.cjs` renames offenders and updates links across other pages in the same pass.

### Why hyphens

GitHub wiki URLs are `<repo>/wiki/Page-Name`. Spaces become `-`. Authoring with `-` from the start avoids:
- Link rot when the title changes capitalization
- Ambiguity between `Page Name.md` (space in filename, unusual on most filesystems) and `Page-Name.md` (hyphenated, URL-identical)

## Internal links

| Form | Use case |
|---|---|
| `[text](Page-Name)` | Link to another wiki page (no `.md`, no leading `./`) |
| `[text](Page-Name#section)` | Link to a section within another page |
| `[text](#section)` | Link within the same page |
| `[text](./images/foo.png)` | Link to an image in the wiki clone's `images/` dir |

GitHub wiki resolves `Page-Name` (no extension) to `Page-Name.md`. Writing `.md` explicitly also works but is inconsistent with the GitHub UI's link generator — prefer no-extension for consistency.

Anchors are generated by GitHub from headings: lowercase, strip punctuation, replace spaces with hyphens. `## Mechanic Registry` → anchor `#mechanic-registry`.

## Linking to non-page assets

Wiki pages can link to non-page assets (data files, source code, plan files, design tables) only via **absolute GitHub URLs**. Relative paths that target a subdirectory the wiki clone doesn't have, or that escape the wiki dir, fail `validate-wiki-links.cjs`.

| Asset lives in | Correct link form |
|---|---|
| Source repo, any path | `https://github.com/<owner>/<repo>/blob/<branch>/<path>` |
| Source repo, raw download | `https://github.com/<owner>/<repo>/raw/<branch>/<path>` |
| Wiki clone `images/` dir | `./images/<file>` (file must be committed to wiki repo separately) |
| Another wiki page | `Page-Name` (no extension, no `./`) |
| External site | full `https://...` URL |

Copy-paste templates:

```
[items.csv](https://github.com/The1Studio/DOTS-AI/blob/master/docs/wiki/data/backpack-crawler/items.csv)
[design plan](https://github.com/The1Studio/DOTS-AI/blob/master/plans/reports/260429-foo.md)
[source ref](https://github.com/The1Studio/DOTS-AI/blob/master/Assets/Demos/Foo/Bar.cs)
```

**Why this matters:** GitHub Wiki is a separate git repo. It does NOT contain the source repo's `data/`, `plans/`, `Assets/`, or any other top-level directories. A link like `./data/items.csv` resolves correctly when viewing the source markdown on github.com/<repo>/blob/master/docs/wiki/ — but breaks immediately on publish to the wiki. `validate-wiki-links.cjs` rejects any link target it can't resolve to a sibling `*.md` file or a wiki-local asset.

**The trap:** authors compose pages in `docs/wiki/<page>.md` where relative paths feel natural and render fine in the source view. The breakage is invisible until publish. Use absolute GitHub URLs from the start to avoid bulk-rewriting later.

See also `github-wiki-gotchas.md` → "Source-repo asset paths don't resolve in wiki" for the deeper diagnostic.

## Frontmatter

Optional but recommended:

```yaml
---
wikiSection: "Game Systems"   # used by sync-sidebar.cjs to group pages
wikiOrder: 10                 # sort order within section (lower = earlier)
wikiStatus: draft             # "draft" hides from published sidebar (if configured)
---
```

Unknown keys are ignored; the validator doesn't reject extensions, only malformed YAML.

## Page structure

```
# <Page Title>            ← single H1 matching the filename (minus hyphens)

Brief one-paragraph intro.

## Section One            ← H2-level sections
### Sub-section           ← H3 for sub-topics; do not skip to H4

Content...

## References             ← optional, last section
- [Related Page](Related-Page)
```

Rules enforced by `validate-section-headers.cjs`:
- Exactly one H1 per page
- No skipped heading levels (no H2 → H4 jumps)
- H1 text should match the filename (hyphens → spaces)

## Sidebar structure

`_Sidebar.md` is a special GitHub-wiki page. It renders as the persistent right-hand nav.

**Canonical style** (adopted 2026-05-27): `<details open>` collapsibles with count suffix, optional status/priority labels, and `🏠 **[Home](Home)**` anchor at top:

```markdown
🏠 **[Home](Home)**

---

<details open>
<summary>⚔️ <b>Core Mechanics · 3</b></summary>

- 📄 [Combat System](Combat-System) P0 ✅
- 📄 [Equipment](Equipment) P0 ✅
- 📄 [Forge System](Forge-System) P0 ✅

</details>

---

> 🔗 [Source repo](https://github.com/owner/repo) — related external resource
```

Full spec — element anatomy (10 elements), two structural patterns (flat list vs bucketed tree), status/priority vocabulary, URL encoding rules, default-state heuristic, AskUserQuestion decision-flow, anti-patterns, migration guide: see [`sidebar-style.md`](./sidebar-style.md).

`sync-sidebar.cjs` regenerates this from:
1. Pages' `wikiSection` frontmatter → super-category bucket
2. Pages' `wikiOrder` → sort within bucket
3. Pages' `status:` / `wikiStatus:` → status icon
4. Fallback grouping: alphabetical

Orphan pages (no `wikiSection`) land under a synthetic `📂 <b>Uncategorized · N</b>` section — the operator should assign them.

**Legacy flat style** (pre-2026-05-27) using `### Heading` + bullets remains parseable by `validate-sidebar.cjs` but is deprecated for new wikis. Wikis with <5 pages MAY remain flat — collapsibles add noise without payoff at that scale.

## Page size

Soft limit: 20KB per page (`validate-chunk-size.cjs`). Larger pages should be split into sub-pages with cross-links. GitHub renders very large wiki pages slowly and anchor-in-page navigation becomes unwieldy.

## Images

Images live in the wiki clone at `images/<filename>`. Reference with `![alt](./images/foo.png)`.

**Not automatically synced from the source repo.** If an image needs to appear in both docs and wiki, copy the file into the wiki clone and commit separately. `validate-images.cjs` flags references to files that don't exist in the wiki clone.

## What NOT to do

- Do not use `Home.md` as a catch-all — keep Home short (navigation landing page, not a dumping ground)
- Do not embed raw HTML — GitHub strips most of it and silently breaks layouts
- Do not nest pages in subdirectories — GitHub flattens paths at render time and breaks links
- Do not hand-edit `_Sidebar.md` heavily — run `fix` (which calls `sync-sidebar.cjs`) after adding/renaming pages
